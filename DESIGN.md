# Herald_MCP — Design Document

> Created: 2026-06-23
> Status: Pre-implementation — approved architecture

---

## 1. Purpose

Herald_MCP enables two Claude instances (or any MCP-capable AI agent) on separate
machines in different locations to communicate directly with each other over a secure
private network. Each machine is a full peer — either side can initiate a conversation.

As a bonus, any machine with just a browser or HTTP client can act as a thin terminal
and use the Claude on either peer machine, without installing any AI software itself
(Plan B, achieved automatically by Plan A).

| Extra auth | None for v1 — Tailscale network-level security is sufficient |

## 3. Network Layer — Tailscale

Tailscale assigns each machine a fixed `100.x.x.x` IP that never changes.
No tunnels, no dynamic URLs, no third-party relay needed.

Machine B joins the Tailscale network using an **auth key** generated by the account
owner on Machine A — no Google account password is shared.

```
Machine A (100.70.161.73)  ←──── Tailscale encrypted P2P ────→  Machine B (100.x.x.x)
```

## 4. Architecture

Each machine runs two processes:

```
┌─────────────────────────────────┐
│           Claude Code            │
│  (or Antigravity / any MCP host) │
└──────────────┬──────────────────┘
               │ MCP stdio
┌──────────────▼──────────────────┐
│         herald mcp_server.py     │  ← MCP layer: exposes tools to Claude
│  Tools: ask_peer / get_pending   │
│          reply / list_peers      │
└──────────────┬──────────────────┘
               │ HTTP (localhost)
┌──────────────▼──────────────────┐
│         herald server.py         │  ← FastAPI HTTP server, port 7700
│  POST /ask                       │     receives questions from remote peer
│  POST /reply/{id}                │     receives replies from local MCP
│  GET  /health                    │     ping / status check
└──────────────┬──────────────────┘
               │ HTTP over Tailscale
          (to remote peer's server.py)
```

Both machines run the **identical** codebase — symmetric design.

## 5. Request-Response Flow

```
Machine A Claude                Herald A           Tailscale         Herald B           Machine B Claude
      │                            │                                     │                      │
      │  ask_peer("B", "message")  │                                     │                      │
      │──────────────────────────→ │                                     │                      │
      │                            │  POST /ask (holds connection open)  │                      │
      │                            │────────────────────────────────────→│                      │
      │                            │                                     │  queues message       │
      │                            │                                     │──────────────────────→│
      │                            │         (waiting…)                  │                      │
      │                            │                                     │  get_pending()        │
      │                            │                                     │←─────────────────────│
      │                            │                                     │  reply(id, answer)    │
      │                            │                                     │←─────────────────────│
      │                            │  HTTP response returns with answer  │                      │
      │                            │←────────────────────────────────────│                      │
      │  ← answer                  │                                     │                      │
      │←──────────────────────────│                                     │                      │
```

Key points:
- Machine A's `ask_peer()` call **blocks and waits** for Machine B's Claude to reply
- Herald B holds the incoming HTTP connection open (long-poll, timeout: 300s)
- Machine B's Claude polls `get_pending()` to discover incoming messages
- Machine B's Claude calls `reply(message_id, answer)` to resolve the waiting request
- If no reply within 300s, the call times out and Machine A's Claude can retry

## 6. File & Image Transfer

Included in v1. Attachments are base64-encoded and embedded in the JSON payload.

| File size | Method |
|-----------|--------|
| < 5 MB | Base64 in JSON payload |
| > 5 MB | Send file path (both machines on same Tailscale network can access shared paths if configured) |

Attachment schema (in both `ask_peer` and `reply`):
```json
"attachments": [
  {
    "filename": "screenshot.png",
    "mime_type": "image/png",
    "data_b64": "iVBORw0KGgo..."
  }
]
```

## 7. MCP Tools (exposed to Claude)

### `ask_peer(peer_name, message, attachments=[])`
Send a question to a named peer's Claude and wait for its reply.
- `peer_name`: key from config (e.g., `"machine-b"`)
- `message`: text prompt
- `attachments`: optional list of files/images
- Returns: `{answer: str, attachments: [...]}`
- Timeout: 300s

### `get_pending()`
Check inbox for incoming questions from remote peers.
- Returns: list of `{message_id, from_peer, message, attachments, received_at}`
- Returns empty list if no pending messages

### `reply(message_id, answer, attachments=[])`
Send a reply to a pending incoming question.
- `message_id`: from `get_pending()`
- `answer`: response text
- `attachments`: optional

### `list_peers()`
Show all configured peers and their online status.
- Returns: `{peer_name: {ip, port, online: bool}}`

### `ping_peer(peer_name)`
Check if a specific peer's herald server is reachable.
- Returns: `{online: bool, latency_ms: int}`

## 8. Configuration (`config.json`)

Each machine has its own `config.json`:

```json
{
  "name": "machine-a",
  "port": 7700,
  "peers": {
    "machine-b": {
      "ip": "100.x.x.x",
      "port": 7700
    }
  }
}
```

Machine B's config swaps the names and IPs. The file is hand-edited once after
Tailscale IPs are known. No dynamic URL updates needed.

## 10. Thin Client (Plan B — free, no extra work)

Once Herald_MCP is running on Machine A, any device on the same Tailscale network
can send HTTP POST requests directly to Machine A's herald server and receive Claude's
response — no AI software required on the client device.

A minimal browser UI (single HTML file) can serve as a full terminal interface.
This is automatically supported by the existing server.py — no additional code needed.

## 12. What This Is NOT

- Not a cloud service — runs entirely on your own machines
- Not dependent on any third-party AI API — uses locally running Claude Code
- Not a replacement for gemi-mcp or agy-mcp — Herald handles machine-to-machine
  routing only; each machine's Claude still uses its own tools independently
